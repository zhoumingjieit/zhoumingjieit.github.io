2.线程相关的参数：
★在线程池内部，其每一个请求由一个线程来处理； 其worker线程的最大数决定了varnish的并发响应能力；
☉thread_pools：
    线程池数量，最好小于或等于CPU核心数量；
☉thread_pool_max：
    每个线程池中的最大线程数量；
☉thread_pool_min：
    每个线程池中的最少线程数量，额外意义为“最大空闲线程数”；
☉最大并发连接数=thread_pools * thread_pool_max
☉thread_pool_timeout：
    每个线程的空闲超时时长，如果超过将被干掉，但不能少于thread_pool_min定义的数量；
☉thread_pool_add_delay：
    在创建线程前的延迟时间
☉thread_pool_destroy_delay：
    在清理线程前的延迟时间

总结：

varnish 扩展：

◆如果单台 varnish 缓存服务器提供的缓存能力，命中率，缓存空间，并发能力等不足以对我们整个网站的规模提供服务，这时就要添加 varnish 主机；但是添加一定要考虑到前端调度器对 varnish 的调度算法；
◆基于缓存命中率的考虑，我们要保证对于同一个 URL的请求要始终发往同一个后端主机，所以我们要基于 URL 的hash 向后端进行调度，这里可取的调度算法有两种，一种是取膜法（任何一个主机down 掉都会使其结果变动，致使所有的缓存项都不再命中），一种是一致性哈希算法（hash $request uri consistent），采用 一致性哈希算法;
◆如果前端调度器为nginx，则基于动态内容请求的服务器，为了绑定会话，要 hash 的是 cookie，即来自于同一个用户的请求始终发往同一台服务器，也使动态内容服务器基于session server 来提供服务；
◆varnish 仅仅是请求内容后用来缓存内容的，因此对于需要维持用户会话的请求千万不能用varnish缓存；
◆一般动态内容之前是不需要缓存的。
